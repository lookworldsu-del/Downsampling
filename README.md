# DownsamplingBenchmark

iOS 实时摄像头采集场景下，CPU vs GPU Downsampling 性能对比工具。

## 项目背景

在移动端图像处理中，Downsampling（降采样）是最常见的操作之一——从相机预览、缩略图生成到视频流预处理都会用到。本项目在 **真实的实时摄像头采集场景** 下，分别实现了 CPU 和 GPU 两大类共 **6 种** Downsampling 方案，并通过全面的性能指标进行实时对比。

## 技术方案

### CPU Downsampling（3 种）

| 方案 | 框架 | 原理 |
|------|------|------|
| **vImage** | Accelerate | 基于 SIMD/NEON 指令集的高性能图像缩放，使用 `vImageScale_ARGB8888` + 高质量重采样 |
| **CGContext** | Core Graphics | 传统 `CGContext.draw(in:)` 双线性插值，作为 CPU 基准对照 |
| **UIGraphicsImageRenderer** | UIKit | UIKit 高层封装，代码最简，适合快速降采样场景 |

### GPU Downsampling（3 种）

| 方案 | 框架 | 原理 |
|------|------|------|
| **Metal Compute** | Metal | 自定义 compute kernel，使用 GPU 硬件纹理采样单元进行双线性插值 |
| **MPS** | MetalPerformanceShaders | Apple 官方 `MPSImageBilinearScale`，GPU 硬件加速最优实现 |
| **Core Image (CILanczos)** | CoreImage | `CILanczosScaleTransform` 滤镜，系统级 GPU 加速，Lanczos 插值质量最高 |

### Metal Shader 核心

```metal
kernel void downsample_bilinear(
    texture2d<float, access::sample> inTexture  [[texture(0)]],
    texture2d<float, access::write>  outTexture [[texture(1)]],
    uint2 gid [[thread_position_in_grid]])
{
    constexpr sampler s(filter::linear, address::clamp_to_edge);
    float2 uv = (float2(gid) + 0.5) / float2(outTexture.get_width(), outTexture.get_height());
    outTexture.write(inTexture.sample(s, uv), gid);
}
```

## 性能指标

采集维度覆盖 **时间、资源、能耗、稳定性** 四大方面：

| 维度 | 指标 | 采集方式 |
|------|------|----------|
| 时间 | 帧处理耗时 (ms) | `mach_absolute_time` 高精度计时 |
| 时间 | 实时 FPS | 1/平均帧耗时 |
| 时间 | P99 耗时 | 滑动窗口排序取 99 分位 |
| 资源 | CPU 占用率 (%) | `thread_basic_info` 遍历进程线程 |
| 资源 | 内存占用 (MB) | `mach_task_basic_info.resident_size` |
| 资源 | GPU 指令耗时 (ms) | Metal `addCompletedHandler` 差值 |
| 能耗 | 电池电量变化 (%) | `UIDevice.batteryLevel` 周期采样 |
| 能耗 | 消耗速率 (%/min) | 电量差 / 运行时长 |
| 能耗 | 热状态 | `ProcessInfo.thermalState` |
| 稳定性 | 丢帧率 | `captureOutput(_:didDrop:)` 计数 |
| 稳定性 | 帧耗时方差 | 衡量处理抖动 |

## 关键知识点

- **AVFoundation 实时采集管线**: CMSampleBuffer → CVPixelBuffer 零拷贝流转
- **Metal Compute Pipeline**: device → library → function → pipelineState → commandBuffer → encoder → dispatch
- **CVMetalTextureCache**: CVPixelBuffer 到 MTLTexture 的高效桥接，避免 CPU-GPU 数据拷贝
- **vImage vs CGContext**: SIMD 加速 vs 通用路径的性能差异
- **MPS vs 自定义 Shader**: Apple 硬件优化内核 vs 手写通用 kernel 的对比
- **iOS 性能监控**: mach_task_info 线程级 CPU 采集、resident_size 内存跟踪、thermalState 降频检测

### Stretch vs Letterbox：直接拉伸 vs 等比填充开销对比

> ⚠️ Stretch 数据为非隔离模式，Letterbox 数据为隔离模式（🔬），直接比数值不严谨。
> 但 **同模式内的排名** 仍有参考价值。

| 方案 | 1080p Stretch | 1080p LB 🔬 | 4K Stretch | 4K LB 🔬 |
|------|-------------|-------------|-----------|-----------|
| **Metal Compute** | 1.67 ms | **5.18 ms** | 2.28 ms | **5.26 ms** |
| vImage | **0.91 ms** | 5.59 ms | 13.69 ms | 13.89 ms |
| CGContext | 3.85 ms | 11.05 ms | 11.28 ms | 12.11 ms |
| UIGraphics | 3.62 ms | 11.08 ms | 10.88 ms | 12.17 ms |
| MPS | 1.43 ms | 13.54 ms | **1.45 ms** | 13.96 ms |
| Core Image | 4.65 ms | 11.33 ms | 9.49 ms | 15.78 ms |

**隔离跑分揭示的关键结论：**

- **Metal Compute 在 Letterbox 下称王**——单 pass `downsample_letterbox` shader 在 1080p 和 4K 下耗时几乎相同（5.18 vs 5.26ms），印证 GPU 处理时间由输出尺寸决定
- **MPS Letterbox 并非 13x 悬崖**——旧数据 18.75ms 包含了非隔离模式下 6 算法叠加的 GPU 争抢；隔离后实际为 13.54ms，仍然较慢但不再是灾难性的
- **CPU 算法（CGContext/UIGraphics）在隔离模式下变慢**——非隔离模式的高 CPU 负载让处理器保持高频，人为压低了耗时；隔离后 CPU 降频到效率核心，真实单任务性能暴露

### 数据分析与结论

> **跑分方法论说明**
>
> 标记 🔬 的 Letterbox 数据采用 **隔离跑分模式**（Isolated Benchmark）：每次只运行 1 个算法 × 60 帧，算法之间冷却 3 秒。这消除了非隔离模式下 6 算法同时运行导致的 CPU 频率偏差、GPU 资源争抢和热降频问题。
>
> 未标记 🔬 的 Stretch / Scale 数据仍为旧的非隔离模式，数值可能存在 1080p 偏快、4K 偏慢的偏差。

**1. 核心结论（416×416 标准基准验证）：CPU 性能取决于输入尺寸，GPU 取决于输出尺寸**

这是整个 benchmark 最重要的发现。416×416 固定输出提供了最纯粹的对照实验——**同一输出、不同输入**：

| 方案 | 1080p → 416²（输入 2M 像素）| 4K → 416²（输入 8.3M 像素）| 增长 |
|------|---------------------------|---------------------------|------|
| **MPS** | 1.43 ms | 1.45 ms | **1.01x（零增长）** |
| Metal Compute | 1.67 ms | 2.28 ms | 1.37x |
| Core Image | 4.65 ms | 9.49 ms | 2.04x |
| CGContext | 3.85 ms | 11.28 ms | 2.93x |
| UIGraphics | 3.62 ms | 10.88 ms | 3.01x |
| **vImage** | **0.91 ms** | **13.69 ms** | **15.0x** |

MPS 耗时从 1.43→1.45ms——**输入像素量增长 4 倍，GPU 耗时完全不变**。因为 GPU 纹理采样单元通过硬件插值完成降采样，工作量仅由输出纹理的线程数（416×416 = 173,056）决定。

vImage 则从 0.91→13.69ms——增长 15 倍。`vImageScale_ARGB8888` 需要遍历所有输入像素进行加权重采样，4K 帧（33.2 MB）远超 A18 的 L2 缓存容量（16 MB），导致 SIMD 流水线频繁 cache miss。

**2. 隔离跑分揭示：非隔离模式的 CPU 频率偏差**

采用隔离跑分后（每次只跑 1 个算法 + 3 秒冷却），结果与非隔离模式出现了显著差异，揭示了两个相反的偏差：

| 方案 | 1080p LB 旧（非隔离） | 1080p LB 新（隔离🔬） | 4K LB 旧（非隔离） | 4K LB 新（隔离🔬） |
|------|---------------------|---------------------|-------------------|-------------------|
| Metal Compute | 2.45 ms | **5.18 ms** | 4.90 ms | **5.26 ms** |
| vImage | 1.87 ms | 5.59 ms | 28.15 ms | **13.89 ms** |
| MPS | 18.75 ms | **13.54 ms** | 18.18 ms | **13.96 ms** |

- **1080p 非隔离偏快**：6 算法并跑使 CPU 维持高频（106% 占用），人为拉高了轻负载算法的成绩
- **4K 非隔离偏慢**：6 算法并跑导致热降频（100%+ 持续），人为压低了重负载算法的成绩
- **隔离模式下 CPU 占用真实暴露**：Metal Compute 仅 16%，vImage 仅 27~47%

**3. Letterbox 模式：Metal Compute 单 pass shader 全面称王**

隔离跑分最重要的发现——Metal Compute 在 Letterbox 下 **1080p 和 4K 耗时几乎相同**：

| 配置 | Metal Compute | 第二名 | 差距 |
|------|-------------|--------|------|
| 1080p → 416² LB | **5.18 ms** | vImage 5.59ms | 1.08x |
| 4K → 416² LB | **5.26 ms** | CGContext 12.11ms | 2.30x |

1080p 和 4K 的 Metal Compute 耗时仅差 0.08ms——因为 `downsample_letterbox` shader 的 GPU 线程数由输出纹理（416×416）决定，与输入分辨率无关。而所有 CPU 算法在 4K 下耗时翻倍。

**4. 没有银弹：冠军随输入分辨率 + 处理模式切换**

10 组数据揭示了清晰的分界线：

- **1080p + Stretch → vImage 称王**：0.91~1.04ms，直接拉伸场景中赢下 3 个配置
- **4K + Stretch → MPS 称王**：1.32~3.27ms，直接拉伸场景中赢下 3 个配置
- **Letterbox（1080p / 4K）→ Metal Compute 称王**：5.18~5.26ms，两个配置全胜
- **4K 大输出 → Core Image 反超**：仅在 4K × 1/2 场景

**5. MPS 的线性扩展规律（Stretch 模式）**

MPS 耗时与输出像素量近似线性关系：

| 输出像素量 | 输出尺寸 | 1080p 耗时 | 4K 耗时 |
|-----------|---------|-----------|---------|
| 3.24 万 | 240×135 | 0.93 ms | — |
| 12.96 万 | 480×270 | 1.45 ms | 1.32 ms |
| **17.31 万** | **416×416** | **1.43 ms** | **1.45 ms** |
| 51.84 万 | 960×540 | 2.82 ms | 3.27 ms |
| 207.36 万 | 1920×1080 | — | 11.19 ms |

Letterbox 隔离跑分下 MPS 耗时 13.54~13.96ms（远超 Stretch 的 1.43ms），多步合成开销（缩放 → 灰色填充 → Blit）仍然是主要瓶颈。

**6. vImage 的 L2 缓存断崖效应**

| 配置 | vImage 排名 | 耗时 | 帧数据大小 vs L2 |
|------|-----------|------|-----------------|
| 1080p → 416² | **第 1** | 0.91 ms | 8.3 MB < 16 MB ✅ |
| 1080p LB🔬 → 416² | 第 2 | 5.59 ms | 8.3 MB < 16 MB ✅ |
| 1080p × 1/4 | **第 1** | 0.98 ms | 8.3 MB < 16 MB ✅ |
| 1080p × 1/2 | **第 1** | 1.04 ms | 8.3 MB < 16 MB ✅ |
| 4K → 416² | **第 6（末位）** | 13.69 ms | 33.2 MB > 16 MB ❌ |
| 4K LB🔬 → 416² | 第 4 | 13.89 ms | 33.2 MB > 16 MB ❌ |
| 4K × 1/8 | 第 6（末位） | 14.31 ms | 33.2 MB > 16 MB ❌ |

隔离跑分揭示 vImage 在 4K Letterbox 下的真实耗时为 13.89ms（非隔离时的 28.15ms 包含了热降频偏差），但仍比 1080p 慢 2.5x，主因仍是 4K 帧超出 L2 缓存。

**7. Core Image 的 JIT 编译问题在 416×416 下最为突出**

| 配置 | Core Image 方差 | 最大耗时 vs 平均 | 解读 |
|------|----------------|-----------------|------|
| 1080p → 416² | **73.06** | 70.29 / 4.65 ms | JIT 首帧编译导致 15x 尖刺 |
| 4K → 416² | 0.37 | 11.19 / 9.49 ms | 高负载摊薄 JIT 开销 |
| 4K LB🔬 → 416² | 10.95 | 35.09 / 15.78 ms | 隔离跑分下方差显著降低 |
| 1080p LB🔬 → 416² | 0.85 | 16.29 / 11.33 ms | 隔离跑分减少了资源争抢 |

**8. 隔离跑分下 CPU 占用率：揭示真实单算法负载**

| 算法 | 1080p LB🔬 CPU | 4K LB🔬 CPU | 解读 |
|------|---------------|-------------|------|
| Metal Compute | **16.3%** | **16.9%** | GPU 主导，CPU 几乎空闲 |
| vImage | 27.1% | 47.3% | 4K 时 CPU 负担明显加重 |
| CGContext | 35.4% | 32.6% | CPU 较均衡 |
| UIGraphics | 31.9% | 33.0% | 与 CGContext 接近 |
| MPS | 33.7% | 33.8% | 多步合成需要 CPU 配合 |
| Core Image | **45.4%** | **59.6%** | GPU 算法中 CPU 占用最高 |

隔离跑分暴露了一个关键事实：非隔离模式下 6 算法叠加导致 CPU 虚高至 100~106%，掩盖了各算法真实的 CPU 负载差异。Metal Compute 仅 16~17% CPU 占用，是 Letterbox 模式下对 CPU 资源最友好的方案。

**9. 内存规律：隔离跑分显著降低峰值内存**

| 输入分辨率 | Stretch 内存 | LB 非隔离内存 | LB 隔离🔬内存 |
|-----------|-------------|--------------|--------------|
| 1080p → 416² | 169~177 MB | 255~264 MB | **165~172 MB** |
| 4K → 416² | 572~604 MB | 558~590 MB | **170~201 MB** |

隔离跑分下峰值内存骤降——因为只有 1 个算法在运行，不需要为 6 个算法同时分配纹理、缓冲区等 GPU 资源。4K 隔离跑分 Metal Compute 峰值仅 170 MB（非隔离时 589 MB），减少了 71%。
