# DownsamplingBenchmark

iOS 实时摄像头采集场景下，CPU vs GPU Downsampling 性能对比工具。

## 项目背景

在移动端图像处理中，Downsampling（降采样）是最常见的操作之一——从相机预览、缩略图生成到视频流预处理都会用到。本项目在 **真实的实时摄像头采集场景** 下，分别实现了 CPU 和 GPU 两大类共 **6 种** Downsampling 方案，并通过全面的性能指标进行实时对比。

## 技术方案

### CPU Downsampling（3 种）

| 方案 | 框架 | 原理 |
|------|------|------|
| **vImage** | Accelerate | 基于 SIMD/NEON 指令集的高性能图像缩放，使用 `vImageScale_ARGB8888` + 高质量重采样 |
| **CGContext** | Core Graphics | 传统 `CGContext.draw(in:)` 双线性插值，作为 CPU 基准对照 |
| **UIGraphicsImageRenderer** | UIKit | UIKit 高层封装，代码最简，适合快速降采样场景 |

### GPU Downsampling（3 种）

| 方案 | 框架 | 原理 |
|------|------|------|
| **Metal Compute** | Metal | 自定义 compute kernel，使用 GPU 硬件纹理采样单元进行双线性插值 |
| **MPS** | MetalPerformanceShaders | Apple 官方 `MPSImageBilinearScale`，GPU 硬件加速最优实现 |
| **Core Image (CILanczos)** | CoreImage | `CILanczosScaleTransform` 滤镜，系统级 GPU 加速，Lanczos 插值质量最高 |

### Metal Shader 核心

```metal
kernel void downsample_bilinear(
    texture2d<float, access::sample> inTexture  [[texture(0)]],
    texture2d<float, access::write>  outTexture [[texture(1)]],
    uint2 gid [[thread_position_in_grid]])
{
    constexpr sampler s(filter::linear, address::clamp_to_edge);
    float2 uv = (float2(gid) + 0.5) / float2(outTexture.get_width(), outTexture.get_height());
    outTexture.write(inTexture.sample(s, uv), gid);
}
```

## 性能指标

采集维度覆盖 **时间、资源、能耗、稳定性** 四大方面：

| 维度 | 指标 | 采集方式 |
|------|------|----------|
| 时间 | 帧处理耗时 (ms) | `mach_absolute_time` 高精度计时 |
| 时间 | 实时 FPS | 1/平均帧耗时 |
| 时间 | P99 耗时 | 滑动窗口排序取 99 分位 |
| 资源 | CPU 占用率 (%) | `thread_basic_info` 遍历进程线程 |
| 资源 | 内存占用 (MB) | `mach_task_basic_info.resident_size` |
| 资源 | GPU 指令耗时 (ms) | Metal `addCompletedHandler` 差值 |
| 能耗 | 电池电量变化 (%) | `UIDevice.batteryLevel` 周期采样 |
| 能耗 | 消耗速率 (%/min) | 电量差 / 运行时长 |
| 能耗 | 热状态 | `ProcessInfo.thermalState` |
| 稳定性 | 丢帧率 | `captureOutput(_:didDrop:)` 计数 |
| 稳定性 | 帧耗时方差 | 衡量处理抖动 |

## 技术栈

- **语言**: Swift 5 / Metal Shading Language
- **UI**: UIKit 纯代码布局（无 Storyboard）
- **摄像头**: AVFoundation (AVCaptureSession + AVCaptureVideoDataOutput)
- **CPU 图像处理**: Accelerate (vImage) / Core Graphics / UIGraphicsImageRenderer
- **GPU 图像处理**: Metal Compute Pipeline / MetalPerformanceShaders / Core Image (CILanczos)
- **性能采集**: mach_task_info / mach_absolute_time / UIDevice Battery API

## 环境要求

- iOS 17.0+
- Xcode 16.0+
- 真机运行（摄像头 + Metal GPU）

1. 用 Xcode 打开 `DownsamplingBenchmark.xcodeproj`
2. 选择真机设备，配置 Signing Team
3. Build & Run
4. 授权摄像头权限后即可看到实时对比

## 关键知识点

- **AVFoundation 实时采集管线**: CMSampleBuffer → CVPixelBuffer 零拷贝流转
- **Metal Compute Pipeline**: device → library → function → pipelineState → commandBuffer → encoder → dispatch
- **CVMetalTextureCache**: CVPixelBuffer 到 MTLTexture 的高效桥接，避免 CPU-GPU 数据拷贝
- **vImage vs CGContext**: SIMD 加速 vs 通用路径的性能差异
- **MPS vs 自定义 Shader**: Apple 硬件优化内核 vs 手写通用 kernel 的对比
- **iOS 性能监控**: mach_task_info 线程级 CPU 采集、resident_size 内存跟踪、thermalState 降频检测

| 方案 | 1080p Stretch | 1080p Letterbox | 开销倍率 | 4K Stretch | 4K Letterbox | 开销倍率 |

| CGContext | 3.85 ms | 5.17 ms | 1.34x | 11.28 ms | 17.99 ms | 1.59x |

关键发现：
- **Metal Compute** 因单 pass shader 处理 letterbox（内框采样 + 外框灰色填充），开销仅 1.5~2x
- **MPS 出现 13x 性能悬崖**——多步合成（MPS 缩放 → CPU 灰色填充 → Blit 拷贝）导致流水线阻塞
- **vImage 稳定 2x 开销**——额外的 CGContext 合成步骤恰好翻倍

### 数据分析与结论

**1. 核心结论（416×416 标准基准验证）：CPU 性能取决于输入尺寸，GPU 取决于输出尺寸**

这是整个 benchmark 最重要的发现。416×416 固定输出提供了最纯粹的对照实验——**同一输出、不同输入**：

| 方案 | 1080p → 416²（输入 2M 像素）| 4K → 416²（输入 8.3M 像素）| 增长 |
|------|---------------------------|---------------------------|------|
| **MPS** | 1.43 ms | 1.45 ms | **1.01x（零增长）** |
| Metal Compute | 1.67 ms | 2.28 ms | 1.37x |
| Core Image | 4.65 ms | 9.49 ms | 2.04x |
| CGContext | 3.85 ms | 11.28 ms | 2.93x |
| UIGraphics | 3.62 ms | 10.88 ms | 3.01x |
| **vImage** | **0.91 ms** | **13.69 ms** | **15.0x** |

MPS 耗时从 1.43→1.45ms——**输入像素量增长 4 倍，GPU 耗时完全不变**。因为 GPU 纹理采样单元通过硬件插值完成降采样，工作量仅由输出纹理的线程数（416×416 = 173,056）决定。

vImage 则从 0.91→13.69ms——增长 15 倍。`vImageScale_ARGB8888` 需要遍历所有输入像素进行加权重采样，4K 帧（33.2 MB）远超 A18 的 L2 缓存容量（16 MB），导致 SIMD 流水线频繁 cache miss。

**2. Letterbox 模式揭示：实现架构决定性能，MPS 出现 13x 性能悬崖**

Letterbox 是 AI 模型推理的实际预处理方式（等比缩放 + 灰色填充保持宽高比）。10 组数据中 Letterbox 提供了最具实践价值的性能画像：

| 方案 | 1080p Stretch | 1080p Letterbox | 开销 | 4K Stretch | 4K Letterbox | 开销 |
|------|-------------|-----------------|------|-----------|--------------|------|
| Metal Compute | 1.67 ms | 2.45 ms | **1.47x** | 2.28 ms | 4.90 ms | **2.15x** |
| Core Image | 4.65 ms | 6.13 ms | 1.32x | 9.49 ms | 16.01 ms | 1.69x |
| CGContext | 3.85 ms | 5.17 ms | 1.34x | 11.28 ms | 17.99 ms | 1.59x |
| UIGraphics | 3.62 ms | 5.22 ms | 1.44x | 10.88 ms | 16.79 ms | 1.54x |
| vImage | 0.91 ms | 1.87 ms | 2.05x | 13.69 ms | 28.15 ms | 2.06x |
| **MPS** | **1.43 ms** | **18.75 ms** | **13.1x ⚠️** | **1.45 ms** | **18.18 ms** | **12.5x ⚠️** |

**Metal Compute 单 pass shader 的架构优势**：`downsample_letterbox` kernel 在一次 GPU dispatch 中完成"内框采样 + 外框灰色填充"，开销仅 1.5~2x。而 MPS 因 API 限制（`MPSImageBilinearScale` 只能写满整个目标纹理），必须走"缩放到临时纹理 → CPU 灰色填充 → Blit 拷贝"三步流水线，CPU↔GPU 同步点导致 13x 性能悬崖。

**4K Letterbox 冠军从 MPS 切换到 Metal Compute**——这是本 benchmark 中最重要的实践结论：当需要自定义后处理（Letterbox、归一化、色彩转换）时，灵活的 Metal Compute Shader 远优于封装好的 MPS。

**3. 没有银弹：冠军随输入分辨率 + 处理模式切换**

10 组数据揭示了四维分界线：

- **1080p + Stretch → vImage 称王**：0.91~1.04ms，直接拉伸场景中赢下 3 个配置
- **4K + Stretch → MPS 称王**：1.32~3.27ms，直接拉伸场景中赢下 3 个配置
- **1080p + Letterbox → vImage 仍然最快**：1.87ms，但 Metal Compute 紧随其后（2.45ms）
- **4K + Letterbox → Metal Compute 称王**：4.90ms，MPS 因多步合成退居第 5
- **4K 大输出 → Core Image 反超**：仅在 4K × 1/2 场景

**4. MPS 的线性扩展规律（仅限 Stretch 模式）**

MPS 耗时与输出像素量近似线性关系：

| 输出像素量 | 输出尺寸 | 1080p 耗时 | 4K 耗时 |
|-----------|---------|-----------|---------|
| 3.24 万 | 240×135 | 0.93 ms | — |
| 12.96 万 | 480×270 | 1.45 ms | 1.32 ms |
| **17.31 万** | **416×416** | **1.43 ms** | **1.45 ms** |
| 51.84 万 | 960×540 | 2.82 ms | 3.27 ms |
| 207.36 万 | 1920×1080 | — | 11.19 ms |

但在 Letterbox 模式下，MPS 的线性规律被多步合成开销完全破坏（416×416 Letterbox 耗时 18ms，远超 1920×1080 Stretch 的 11ms）。

**5. vImage 的 L2 缓存断崖效应（Letterbox 进一步放大）**

| 配置 | vImage 排名 | 耗时 | 帧数据大小 vs L2 |
|------|-----------|------|-----------------|
| 1080p → 416² | **第 1** | 0.91 ms | 8.3 MB < 16 MB ✅ |
| 1080p LB → 416² | **第 1** | 1.87 ms | 8.3 MB < 16 MB ✅ |
| 1080p × 1/4 | **第 1** | 0.98 ms | 8.3 MB < 16 MB ✅ |
| 1080p × 1/2 | **第 1** | 1.04 ms | 8.3 MB < 16 MB ✅ |
| 4K → 416² | **第 6（末位）** | 13.69 ms | 33.2 MB > 16 MB ❌ |
| 4K LB → 416² | **第 6（末位）** | 28.15 ms | 33.2 MB > 16 MB ❌ |
| 4K × 1/8 | 第 6（末位） | 14.31 ms | 33.2 MB > 16 MB ❌ |

vImage 在 4K Letterbox 下耗时达 28.15ms（Stretch 的 2x），因为 Letterbox 需要额外的 CGContext 合成步骤，而 4K 帧已经超出 L2 缓存，合成操作进一步加剧 cache miss。

**6. Core Image 的 JIT 编译问题在 416×416 下最为突出**

| 配置 | Core Image 方差 | 最大耗时 vs 平均 | 解读 |
|------|----------------|-----------------|------|
| 1080p → 416² | **73.06** | 70.29 / 4.65 ms | JIT 首帧编译导致 15x 尖刺 |
| 4K → 416² | 0.37 | 11.19 / 9.49 ms | 高负载摊薄 JIT 开销 |
| 4K LB → 416² | **79.02** | 84.00 / 16.01 ms | Letterbox 滤镜链触发二次 JIT |
| 1080p LB → 416² | 0.58 | 10.58 / 6.13 ms | 较轻负载但复合滤镜更稳定 |

**7. CPU 占用率：Letterbox 模式推高至 100%+**

| 输入分辨率 | Stretch | Letterbox | 解读 |
|-----------|---------|-----------|------|
| 1080p | 45.4% ~ 45.5% | **106.1% ~ 106.3%** | Letterbox 合成占用第二核 |
| 4K | 88.8% ~ 89.4% | **100.1% ~ 100.4%** | 已饱和，反而趋于一致 |

Letterbox 模式下 1080p 的 CPU 占用率从 45% 飙升至 106%——合成步骤（灰色填充 + draw）将工作量扩展到多核。4K 原本已接近满载，Letterbox 额外开销使其稳定在 100%。

**8. 内存规律：Letterbox 略高于 Stretch**

| 输入分辨率 | Stretch 内存范围 | Letterbox 内存范围 | 增量 |
|-----------|-----------------|-------------------|------|
| 1080p → 416² | 169 ~ 177 MB | 255 ~ 264 MB | +86 MB |
| 4K → 416² | 572 ~ 604 MB | 558 ~ 590 MB | ~持平 |

1080p Letterbox 内存增加约 86 MB，主要来自中间缩放缓冲区（vImage 需要额外分配内框尺寸 buffer）和合成画布。4K 场景下内存被输入帧缓冲区主导，Letterbox 增量可忽略。

### 方案选择建议

| 场景 | 推荐方案 | 理由 |
